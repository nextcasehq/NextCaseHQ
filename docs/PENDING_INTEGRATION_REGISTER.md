## Resolved: Former Critical Production Blocker (Admin Panel Authentication)

**Resolved by PR #65 (commit `4b41b98`).** The admin console's authentication used to be entirely client-side (`apps/web/src/app/admin/page.tsx` checked a hardcoded password — one valid value was literally the word `admin` — and wrote a hardcoded `NEXTCASE_ADMIN_TOKEN` cookie value itself). It is now server-verified: `POST /api/admin/session` checks the submitted key against `ADMIN_ACCESS_TOKEN` (constant-time, rate-limited, origin-checked) and mints a signed, httpOnly, Secure, SameSite=Strict session cookie; `middleware.ts`'s `/api/admin/*` gate verifies that signed token. The admin UI's markup/labels/tabs are unchanged — only the login/logout/session-check wiring changed. See `known_limitations.admin_auth_shared_secret_no_individual_accountability` in `docs/BUILD_LEDGER.json` for the one tradeoff this milestone deliberately left open (a single shared operator secret, not per-admin accounts).

## ⚠ Observed: Flaky Redis-Backed Rate-Limit Tests Under Parallel Jest Workers

Observed on PR #86's CI run (`test` job, run `29652015160`): `src/app/api/auth/session/__tests__/route.test.ts`'s "rate-limits repeated login attempts" test failed once (expected 429, got 401), then passed cleanly on an immediate re-run of the same job with no code changes. This PR's diff touches only marketing-page JSX/Tailwind classes — no server, auth, or Redis code — so this is not a regression from this PR. Locally, the same class of flake was reproduced in a different rate-limit test (`admin/session`) when running the full suite with Jest's default parallel workers, and confirmed to pass reliably both in isolation and with `--runInBand` after a `redis-cli FLUSHALL`. Root cause: multiple test files whose rate-limit tests hit the same real, shared Redis instance (per `docs/ci.yml`'s `REDIS_URL: redis://localhost:6379` service) are not isolated from each other when Jest runs them in parallel workers — a pre-existing test-infrastructure gap, not something introduced by this PR. Not fixed here (would be unrelated-to-this-PR test-infra work); flagged for a future dedicated fix — e.g. a `beforeEach`/`afterEach` Redis key flush scoped per test file, or per-worker Redis DB indexes (`ioredis` supports `db: <n>` in its options) — rather than continuing to rely on re-running CI when this flakes.

## ⚠ Decision Needed: Webhook Rate Limit / Replay Protection on Edge Runtime

`POST /api/webhooks` runs on the Edge runtime (`export const runtime = 'edge'`), where `ioredis` (a raw-TCP client) cannot run — the Edge runtime has no `net`/`tls` modules. DB-7 above closed the shared-state gap for the two Node.js-runtime routes only; the webhook route's rate limit and replay guard (`lib/security/rate-limit.ts`, `lib/security/replay-guard.ts`) are still in-memory/per-instance. Two real options, neither chosen unilaterally:
- **(A)** Adopt an Edge-compatible REST-based Redis client (e.g. Upstash Redis or Vercel KV) for this route specifically — a concrete vendor/account choice.
- **(B)** Move `POST /api/webhooks` off the Edge runtime to Node.js runtime so the same `ioredis`-backed limiter can cover it too — an architecture change to a route explicitly built as Module 13 ("Edge Webhook Byte Handling").

Awaiting Product Owner decision before implementing either.

## ⚠ Technical Debt: Pre-Existing Shared-Tenant-ID Test Race Under Parallel Jest Workers

Observed while stress-testing PR #95 (Court Note Quick Entry Foundation) — not a regression introduced by that PR, reproduced identically against the unmodified pre-PR codebase.

- **Affected test files** (all three hardcode the same tenant UUID `00000000-0000-4000-8000-0000000000d1` — and, for two of them, the same `00000000-0000-4000-8000-0000000000d2` — as a shared, pre-existing convention):
  - `apps/web/src/app/api/cases/[id]/__tests__/route.test.ts`
  - `apps/web/src/lib/db/__tests__/matter-rls.test.ts`
  - `apps/web/src/app/api/auth/session/__tests__/route.test.ts`
- **Shared tenant identifier:** `TENANT_A = '00000000-0000-4000-8000-0000000000d1'` (also used as `TENANT_ID` in the session test); `TENANT_B = '00000000-0000-4000-8000-0000000000d2'` in the other two.
- **Reproduction evidence:** `apps/web/src/app/api/auth/session/__tests__/route.test.ts`'s `afterAll` calls `db.executeSystem('DELETE FROM "Tenant" WHERE id = $1', [TENANT_ID])` — an RLS-bypassing delete of the shared Tenant row itself. When Jest schedules that file's worker concurrently with `matter-rls.test.ts` or `cases/[id]/__tests__/route.test.ts` (both still using the same tenant id), the delete can fire mid-run in the other file, producing a foreign-key violation (`Matter_tenant_id_fkey` / `LegalCase_tenant_id_fkey`) in whichever test happens to be inserting at that moment. Reproduced 1-in-3 on a completely fresh, freshly-migrated local Postgres instance running the full suite via plain `pnpm --filter web test` (default parallel workers) — confirmed via `git stash -u` back to the pre-PR-#95 commit and re-running the identical command three times with a dropped/recreated database between each run. Not reproducible when running just these three files in isolation (too few workers/too fast for the race window to open) — only appears under the full 71-suite parallel run.
- **Impact:** Intermittent, order-of-scheduling-dependent CI/local test failures unrelated to the diff of whatever PR happens to trigger the unlucky worker interleaving — a false-negative risk that costs a re-run and an investigation each time it fires, and could mask a real regression landing in the same run by drawing attention to the wrong failure.
- **Recommended isolated correction (not implemented, not approved):** give each of the three files its own genuinely unique tenant/user UUID (the fix already applied to the two new Milestone-1 test files added in PR #95, which deliberately avoided this shared id), removing the shared-id convention entirely rather than patching the race's symptom; alternatively, scope `session/route.test.ts`'s `afterAll` cleanup to not use `executeSystem`'s RLS-bypassing delete for a tenant id other test files may still be actively using. Do not implement either without separate Product Owner/engineering approval — this is a test-infrastructure change spanning three files outside this PR's scope.
