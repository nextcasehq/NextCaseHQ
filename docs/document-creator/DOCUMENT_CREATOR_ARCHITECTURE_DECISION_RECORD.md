# Document Creator Architecture Decision Record

Status: **Approved, Phase 0 — Architecture Decision Record**
Scope: Audits the exact existing infrastructure the Document Creator will reuse, the gaps against the target architecture described in `DOCUMENT_CREATOR_IMPLEMENTATION_ROADMAP.md`, and the decisions that require explicit Product Owner approval before Phase 4 onward.

## Summary of Genuine Gaps (Not Yet Built)

1. **`DocumentGenerationJob`** — does not exist. New table, new migration.
2. **Real AI Credit reservation/debit call sites** — the entitlement checkpoint exists but is a pass-through; the real wallet exists but nothing calls it from the AI pipeline.
3. **Asynchronous worker execution** — generation is currently synchronous and inline in the web request.
4. **Durable, independently-retryable indexing** — indexing exists and works, but is not yet a resumable job with its own retry semantics.
5. **Continuous autosave with local + server recovery** — does not exist; drafting today is entirely in-memory React state with a "Save Draft" prototype action that shows an honest "not yet available" prompt.
6. **A `DocumentAuditEvent`-equivalent for generation-specific events** — decision pending (see below).

## Decisions Requiring Product Owner Approval

1. **Worker platform selection** (Phase 5) — deliberately left open in `EPHEMERAL_WORKER_ARCHITECTURE.md`; requires a named decision before Phase 5 implementation begins.
2. **New `DocumentAuditEvent` table vs. extending `MatterAuditEvent`** — extending risks conflating two different audit domains (Matter lifecycle vs. document generation lifecycle) under one table's meaning; a new table risks duplicating the append-only/RLS boilerplate. Recommend a new table following the identical pattern, but this is a Product Owner call, not a default.
3. **AI Credit reservation representation** — whether a reservation is a new `status`/`type` value on the existing `WalletTransactionRecord.type` column, or a new dedicated table. Affects the `TenantWallet` schema and is a real, reviewable migration decision, not an implementation detail.
4. **Template storage model** — where "approved templates" (Cold Memory, per `DOCUMENT_CREATOR_MEMORY_MODEL.md`) actually live once they move beyond the current inline-array prototype in `draft-builder`. Not yet designed; needs its own decision before Phase 2/3 touches templates.

None of these four are resolved by this document — they are named explicitly so Phase 2 onward does not resolve them silently in code, per `DOCUMENT_CREATOR_GOVERNANCE.md` §10.
